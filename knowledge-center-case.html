Case 03

熊猫知识中心 + 模型选型助手

面向开发者的模型接入与选型决策平台。围绕模型选择困难、信息分散和客服重复答疑问题，设计真实知识中心、模型选型助手、信息架构和选型规则。

结构化 模型信息、价格、场景和 API 接入内容
自助推荐 从人工咨询转向规则化选型
决策效率 提升模型选择与场景匹配效率
Outputs

实际产出总览

项目三定位为真实知识中心 + 模型选型助手 + 信息架构与选型规则设计。

知识中心首页提供搜索、导航、精选内容和高频报错入口。
FAQ 内容体系覆盖个人用户、企业用户、新手入门、接入教程和报错处理。
教程 / 技巧文章沉淀 API 接入、工具配置、Token 节省、用量监控和成本优化内容。
模型选型助手基于任务场景、效果偏好、速度、成本、稳定性等维度辅助用户选择模型。
模型策略内容整理模型能力、适用场景、价格提醒和使用建议。
报错排查路径将 401、403、429、500、invalid beta flag、messages is null 等高频问题结构化处理。

FAQ 页面采用左侧分类导航、正文问答、右侧目录的结构，承接账号充值、API 接入、计费说明、报错处理等高频问题，降低重复咨询成本。

报错页面把 401 等高频问题拆成错误原因、解决方法、注意事项和上下篇导航，帮助用户自助排查 API Key、权限和配置问题。
Model Picker

真实模型选型助手截图

模型选型助手用于把“我该用哪个模型”的模糊问题转化为结构化判断。用户先选择任务场景，再补充效果、速度、成本、稳定性和推荐方式等偏好，系统据此生成推荐方案。

模型选型助手输入页

选型助手将用户需求拆解为任务场景、偏好权重、稳定性要求和推荐方式，例如代码开发、企业知识库、客服机器人、数据分析、AI Agent 等场景。

模型选型助手结果页

结果页输出主推方案、备选方案、匹配分、能力标签和分析理由，让模型推荐从“给答案”变成可解释的选型判断。
Product Value

从文档站到知识服务系统

项目三的核心不是简单堆文档，而是把用户在模型接入、工具配置、价格理解、报错排查和模型选择中的高频问题，组织成可搜索、可导航、可推荐、可持续更新的知识服务系统。

路径清晰从新手上手、工具接入、常见问题到报错排查，按用户使用路径组织内容。
问题可搜索用户可以直接搜索 API Key、Claude 原生协议、Cursor 接入、Token 计费、错误码等关键词。
内容可复用FAQ、教程、技巧文章和模型策略可以沉淀客服与运营的重复答疑经验。
选型可解释模型选型助手不是直接给模型名，而是基于场景、偏好、稳定性和成本输出推荐依据。
信息可维护模型能力、价格、协议和工具接入方式变化较快，因此内容结构要便于后续更新。

项目背景

PandaToken 接入的模型和渠道不断增加，用户在选择模型时容易遇到信息分散、参数复杂、场景匹配困难的问题。因此设计熊猫知识中心和模型选型助手，帮助用户快速理解模型差异并完成选型。

我的角色

  • 模型信息结构化整理
  • 知识中心信息架构设计
  • 模型选型规则设计
  • 推荐逻辑说明、页面内容整理
  • FAQ / 使用指南设计
Contribution

我具体做了什么

  • 将模型信息按能力、价格、上下文长度、速度、稳定性和适用场景结构化。
  • 设计知识中心的信息架构，包括模型介绍、场景推荐、API 接入指南、FAQ 和最佳实践。
  • 设计模型选型助手规则，根据任务类型、成本预算、响应速度、上下文长度、稳定性等条件推荐模型。
  • 为推荐结果设计“推荐模型 + 推荐理由 + 备选方案”的表达方式。
  • 整理页面内容和用户使用说明，降低客服重复答疑成本。
Problem

核心问题

能力差异不清晰用户不了解不同模型在文本、代码、长文本、多模态等场景中的能力差异。
信息分散模型价格、上下文长度、速度、稳定性等信息缺少统一入口。
重复答疑客服和运营需要反复回答类似问题。
路径不明确新用户缺少清晰的模型选择路径。

Rules

模型选型助手规则

规则不是替用户做绝对判断，而是把用户决策路径结构化，让推荐结果有明确理由。

问题 1：你要做什么？

代码开发企业知识库客服机器人数据分析AI AgentAI 漫剧生图生视频配音音频向量检索

问题 2：你最看重什么？

低成本速度快效果好中文强长上下文稳定工具调用角色一致性图生视频

问题 3：要稳定吗？

企业生产环境需要高并发

问题 4：看哪种结果？

自动判断单模型推荐组合方案推荐
Recommendation Logic

推荐结果结构

即使当前真实截图主要展示选型输入页，作品集中也需要说明推荐结果页如何结构化输出。

主推模型基于当前任务场景和偏好权重给出优先推荐。
备选模型提供成本、速度或稳定性不同取向的备选方案。
推荐理由说明推荐依据，避免只给模型名。
价格提醒基于当期价格表 / 官方文档整理成本提示，避免写死长期不变结论。
能力标签标注代码、长上下文、中文、工具调用、多模态等能力。
接入路径指向相关工具接入教程或 API 配置说明。
注意事项提示稳定性、并发、协议、成本或更新频率相关风险。

关键成果

  • 搭建模型信息结构化展示框架。
  • 将模型选择从人工咨询转化为自助推荐。
  • 沉淀模型选型规则和知识中心内容框架。
  • 提升用户在模型选择、API 接入和场景匹配中的决策效率。

项目复盘

知识库不是资料堆积，而是要把用户决策路径结构化，让用户更快完成选择。这个项目让我更关注信息架构、规则解释和用户自助路径之间的关系。
